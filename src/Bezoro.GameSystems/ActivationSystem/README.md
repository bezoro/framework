# ActivationSystem

ECS-native activation pipeline for deferred callback execution with priority ordering and per-tick budget control.

## Types

| Type                            | Description                                                                |
|---------------------------------|----------------------------------------------------------------------------|
| `ActivationCommandQueue`        | Thread-safe ingress queue for `Register` and `Cancel` commands.            |
| `ActivationHandle`              | Stable handle used to identify and cancel pending activation entries.      |
| `ActivationEntry`               | ECS component storing callback, priority, handle, and `ActivationState`.   |
| `ActivationCancellationRequest` | ECS request component generated by ingestion for cancellations.            |
| `ActivationConfig`              | Resource configuring per-tick budget and callback dispatcher.              |
| `ActivationRuntimeState`        | Resource exposing runtime counts (`PendingCount`, `ActivatedCount`).       |
| `ActivationCompletedEvent`      | Event payload emitted when pending activation work completes.              |
| `ActivationEventsResource`      | Pull-based queue of `ActivationCompletedEvent` values.                     |
| `ActivationIngestionSystem`     | `Input`-stage system that drains the command queue into ECS requests/data. |
| `ActivationProcessingSystem`    | `Tick`-stage system that applies cancellation and activates pending work.  |
| `ActivationDispatchSystem`      | `PostTick`-stage system that invokes activated callbacks.                  |
| `ActivationWorldExtensions`     | Pipeline wiring and resource helper extensions for `World`.                |

## Quick Start

```csharp
using Bezoro.ECS.Services;
using Bezoro.GameSystems.ActivationSystem.Extensions;
using Bezoro.GameSystems.ActivationSystem.Types;

var world = new World();
world.SetResource(new ActivationConfig(maxActivationsPerTick: 4));
world.AddActivationPipeline();

var queue = world.GetOrCreateActivationCommandQueue();

var handleA = queue.Register(() => LoadChunk(0, 0), priority: 10);
var handleB = queue.Register(() => LoadChunk(1, 0), priority: 5);
var handleC = queue.Register(() => LoadChunk(0, 1), priority: 5);

queue.Cancel(handleC);

world.Tick(0f);

var runtime = world.GetResource<ActivationRuntimeState>();
Console.WriteLine($"Pending: {runtime.PendingCount}");
Console.WriteLine($"Activated: {runtime.ActivatedCount}");
Console.WriteLine($"Complete: {runtime.IsComplete}");
```

## Unity Integration

Use activation as a frame-budgeted pipeline: instantiate/pool inactive objects, queue activation callbacks, and process a limited number per `Update`.

```csharp
using System.Threading;
using Bezoro.ECS.Services;
using Bezoro.GameSystems.ActivationSystem.Extensions;
using Bezoro.GameSystems.ActivationSystem.Types;
using UnityEngine;

public sealed class UnityActivationBootstrap : MonoBehaviour
{
	private World _world = null!;
	private ActivationCommandQueue _queue = null!;
	private SynchronizationContext _unityContext = null!;

	[SerializeField] private int maxActivationsPerFrame = 4;

	private void Awake()
	{
		_unityContext = SynchronizationContext.Current!;
		_world = new World();

		_world.SetResource(new ActivationConfig(
			maxActivationsPerTick: maxActivationsPerFrame,
			callbackDispatcher: callback => _unityContext.Post(_ => callback(), null)
		));

		_world.AddActivationPipeline();
		_queue = _world.GetOrCreateActivationCommandQueue();
	}

	private void Update()
	{
		_world.Tick(Time.deltaTime);
	}

	public ActivationHandle QueueGameObjectActivation(GameObject gameObject, int priority = 0)
	{
		gameObject.SetActive(false);
		return _queue.Register(() =>
			{
				if (gameObject != null)
					gameObject.SetActive(true);
			},
			priority
		);
	}

	public bool CancelActivation(ActivationHandle handle) => _queue.Cancel(handle);

	private void OnDestroy() => _world.Dispose();
}
```

This keeps Unity API calls on the main thread while spreading activation work across frames.

## API Reference

### `ActivationCommandQueue`

| Member                                        | Description                                             |
|-----------------------------------------------|---------------------------------------------------------|
| `Register(Action callback, int priority = 0)` | Queues activation and returns a new handle.             |
| `Cancel(ActivationHandle)`                    | Queues cancellation for a pending handle.               |
| `PendingCommandCount`                         | Approximate queued command count waiting for ingestion. |

### `ActivationConfig`

| Member                  | Description                                                           |
|-------------------------|-----------------------------------------------------------------------|
| `MaxActivationsPerTick` | Maximum entries activated by processing each tick (`<= 0` means all). |
| `CallbackDispatcher`    | Optional dispatcher used by `ActivationDispatchSystem` for callbacks. |

### `ActivationRuntimeState`

| Member           | Description                                          |
|------------------|------------------------------------------------------|
| `PendingCount`   | Current number of pending activation entries.        |
| `ActivatedCount` | Current number of activated entries.                 |
| `IsComplete`     | `true` when there are no pending activation entries. |

### `ActivationWorldExtensions`

| Method                                     | Description                                                     |
|--------------------------------------------|-----------------------------------------------------------------|
| `GetOrCreateActivationCommandQueue(World)` | Returns existing queue resource or creates one.                 |
| `AddActivationPipeline(World)`             | Registers ingestion, processing, and dispatch systems in order. |

## Design Notes

- External threads only interact with `ActivationCommandQueue`.
- ECS systems perform all state mutation on the world update thread.
- Priority ordering is deterministic: higher priority first, then lower handle id.
- Completion is edge-triggered and published through `ActivationEventsResource`.
- Callback and event exceptions are isolated and do not break simulation.
